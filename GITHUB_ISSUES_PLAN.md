# GitHub Issues Plan

## Objetivo
Transformar o backlog da v1 em issues rastreaveis no GitHub.

## Estrutura recomendada
Criar issues separadas para:
- infraestrutura Firebase
- importacao de conteudo
- setup do FlutterFlow
- paginas principais
- modulo Ajuste Inicial
- modulo Assincronias
- modulo Casos e Quiz
- SimulatorPage
- ProgressPage e FeedbackPage
- AdminPage
- testes PWA
- publicacao piloto

## Regra de granularidade
Cada issue deve representar uma entrega observavel e testavel.
Evitar issues grandes demais para nao perder rastreabilidade.

## Labels sugeridas
- p0
- p1
- p2
- frontend
- backend
- content
- simulator
- admin
- testing
- release

## Fluxo recomendado
1. Abrir milestones da v1
2. Criar issues por sprint
3. Associar labels e milestone
4. Fechar issues apenas quando o criterio de pronto estiver atendido

## Boa pratica
Sempre relacionar bug ao modulo afetado e ao fluxo de reproducao.
